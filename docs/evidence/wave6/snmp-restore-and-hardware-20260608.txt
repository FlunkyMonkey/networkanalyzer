ROOT-CAUSE FINDING (pre-existing outage, not new Wave 6 work)
------------------------------------------------------------
The deployed snmp-exporter ConfigMap carried the literal placeholder community
"SNMP_COMMUNITY". The MikroTik silently dropped the bogus-community walk -> request
timeout -> Prometheus scrape target down (up=0/1) -> ZERO interface metrics
(ifHCInOctets, ifSpeed, anything if*). The "working" switch bandwidth dashboards
were actually dark. Almost certainly went dark when an ArgoCD sync reasserted the
git placeholder over a manual fix. Nothing surfaced it.

Fix: community set to "public" (the switch's read-only default; non-sensitive).

ITEM 4 — Switch Interface Utilization % gauge
---------------------------------------------
Root issue was twofold: (a) no SNMP data at all (community), (b) 32-bit ifSpeed
overflows to 0 on 10G SFP+. Both addressed:
- ifHighSpeed (64-bit Mbps, ifXTable) added with ifDescr+ifAlias lookups.
- Gauge denominator changed from clamp_min(ifSpeed,1) to (ifHighSpeed * 1e6 > 0),
  which is 10G-safe and drops logical/zero-speed interfaces (bridge1).
Live ifHighSpeed for active host ports (all 1Gb):
  ether7  prox1-eno1     1000 Mb
  ether6  prox2-enp1s0f0 1000 Mb
  ether19 prox3-enp1s0f1 1000 Mb
  ether23 prox4-enp3s0   1000 Mb
  ether3  prox5-enp3s0   1000 Mb
(Only bridge1 has ifHighSpeed=0 - logical aggregate, correctly excluded.)
